Managing dependencies across multiple repositories was challenging and time-consuming. In a project at a previous job, we had four private GitHub repositories, with one serving as a shared package called "shared-components". This package was published to NPM and installed as a dependency in the other projects.

Having one centralized place for the code was beneficial for consistency and reducing duplicated code. But the process was slow. Each time we published a new version to NPM, we had to create three separate merge requests for the other repositories, which could take up to ten minutes per update.

## Github Action

To improve this process, we implemented a GitHub Action that automated the update procedure. This action:

1. Retrieved the current version of "shared-components" from NPM
2. Installed the new version
3. Committed the changes to a new branch
4. Created a merge request against the main branch if there were differences

```yaml
name: Update Shared Components

on:
  workflow_dispatch:

jobs:
  update-package:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: "18.20.4"

      - name: Get latest version of shared-components
        id: get-version
        env:
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
        run: |
          echo "Fetching latest version of @company/shared-components"
          LATEST_VERSION=$(npm view @company/shared-components version)
          echo "version=$LATEST_VERSION" >> $GITHUB_OUTPUT

      - name: Update package.json
        id: update-package-json
        env:
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
        run: |
          LATEST_VERSION=${{ steps.get-version.outputs.version }}
          TIMESTAMP=$(date +%s)
          BRANCH_NAME="update-shared-components-${LATEST_VERSION}-${TIMESTAMP}"
          echo "Updating package.json with version $LATEST_VERSION"
          npm install @company/shared-components@$LATEST_VERSION
          git config --global user.name 'github-actions[bot]'
          git config --global user.email 'github-actions[bot]@users.noreply.github.com'
          git add package.json package-lock.json
          git commit -m "chore: shared-components bump package to $LATEST_VERSION"
          echo "branch_name=$BRANCH_NAME" >> $GITHUB_OUTPUT

      - name: Create Pull Request
        uses: peter-evans/create-pull-request@v6
        with:
          token: ${{ secrets.GITHUB_TOKEN }}
          branch: ${{ steps.update-package-json.outputs.branch_name }}
          commit-message: "chore: shared-components bump package to ${{ steps.get-version.outputs.version }}"
          title: "chore: shared-components bump package to ${{ steps.get-version.outputs.version }}"
          body: |
            This PR updates the @company/shared-components package to version ${{ steps.get-version.outputs.version }}

            Auto-generated by [create-pull-request][1]

            [1]: https://github.com/peter-evans/create-pull-request
          base: main
```

This automation significantly reduced the time and effort required to keep all repositories up to date with the latest shared components.

To implement this solution, created a new workflow file in the `.github/workflows` directory of each project, naming it appropriately (e.g., `create-merge-request.yml`).

One key component of the GitHub Action was the use of `peter-evans/create-pull-request`. This action automated the process of creating pull requests, making it well suited to our use case. It allowed us to programmatically create pull requests with the updated shared components, saving time and reducing the potential for human error.

Here's a breakdown of how the workflow worked:

1. Trigger: The workflow was configured to run via workflow_dispatch, allowing it to be triggered manually when needed.
2. Node.js Setup: We used Node.js version 18.20.4 to ensure compatibility with our project requirements.
3. Version Fetching: The workflow fetched the latest version of the shared-components package from NPM.
4. Package Update: It updated the package.json and package-lock.json files with the new version.
5. Git Configuration: The workflow configured Git using the github-actions[bot] identity.
6. Branch Naming: A unique branch name was generated using the version number and a timestamp to avoid naming conflicts.
7. Pull Request Creation: The peter-evans/create-pull-request action created a pull request with a descriptive title and body targeting the main branch.
8. Security: The workflow used GitHub secrets (GITHUB_TOKEN and NPM_TOKEN) to securely authenticate with GitHub and NPM.

> 💡 Note: Because our NPM package was private, we had to configure an NPM_TOKEN secret in each repository so the workflow could access the private registry.

By automating dependency updates with GitHub Actions, we reduced repetitive maintenance work and spent less time manually synchronizing repositories.

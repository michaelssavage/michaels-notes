### Overview

A Django REST Framework app that scrapes movie data from Letterboxd and stores it in PostgreSQL. The project started in Go (Gin + MongoDB, deployed on fly.io), then moved to Python to match day-to-day work and make use of Django’s tooling.

It still does the same job: pull favorite films, clean the data, and expose it through an API that can be consumed elsewhere.

### Key Features

1. **Letterboxd scraping**  
  Extracts favorite movies and related metadata
  Handles pagination and repeated fetches
2. **Image processing**  
  Cleans image URLs (removes query params)
  Resizes images to usable resolutions
  Stores final URLs in the database
3. **REST API endpoints**  
  Health check
  Scrape favorites
  Retrieve saved movies
  Save movies manually
4. **Extended endpoint**  
  Save a movie by title via POST request (/movies/save/)
  Example: `{"movie_title":"apocalypse-now"}`
5. **Simple data model**  
  Title, year, status, image URL, link URL
  Timestamps for tracking changes

### Implementation

The project was rebuilt using Django REST Framework after an initial version in Go. I moved to Python to take advantage of the Django ecosystem as my new job role would use this stack. I previously wrote about it here <Anchor link="/blog/rest-api-with-go" text="REST API with Go" variant="link" />. The earlier setup used Gin, MongoDB, and fly.io, with a Netlify function acting as a bridge to display results. The structure there followed a clean separation (controller, service, repository), which worked well, but the switch to Django introduced a more opinionated approach with less setup overhead.

Scraping focuses on extracting movie data from Letterboxd and normalizing it before storage. Image URLs are cleaned to remove unnecessary query strings, then adjusted to point to higher-resolution versions. This step avoids storing low-quality assets while keeping the dataset small and consistent.

The API exposes a set of endpoints for scraping, retrieving, and saving movies. On top of the base endpoints, a dedicated route allows saving individual movies by title. This makes it possible to extend the dataset without re-running full scraping jobs.

Data is stored in PostgreSQL with a minimal schema. Each movie is tied to a status (FAVORITE or SAVED), and a unique entry is maintained per record. Timestamps track when entries are created and updated.

Dependency and environment management is handled with <Anchor link='https://github.com/astral-sh/uv' text='uv' variant='link' isExternal />, which simplifies virtual environments and dependency installation. The app is deployed on Railway using a Procfile-based setup, with basic security headers enforcing HTTPS access.
